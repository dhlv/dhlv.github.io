French-1

Posted on 2020-02-01 | In Language

Lession 1

  • bon → good.

  • c’est → It’s.

    C’est bon.

  • fantastique → fantastic.

  • absolument → absolutely.

    C’est absolument fantastique.

  • ici → here.

  • très → very.

    C’est très bon ici.

  • ce n’est pas → It isn’t.

    Ce n’est pas bon ici.

  • délicieux → delicious.

    è = “fed”.
    é = “pay”.

  • extraordinaire → extraordinary.

  • beau → beautiful.

  • pour moi → for me.

    If put at the beginning of the sentence, means “I think”.
    - Pour moi, c’est trè délicieux.

  • mais → but.

    In French, only pronounce letters “C, R, F, L” in the end of a word.

  • oui → yes.

  • non → no.

  • merci → thanks.

  • toujours → always.

  • parfait → perfect.

    Ce n’est pas toujours parfait ici.

  • ça → that.

    “ç” pronouns as “s”.
    c + e/i = s.
    c+ a/o/u = k.(café)

  • ça c’est → that is.

  • pour lui/elle → for him/her.

  • mauvais → bad.

  • le restaurant → the restaurant.

  • n’est pas → isn’t

  • tout → everyting

    Tout est très bon.

Intro to Docker

Posted on 2020-02-01 | In DevOps

What is Docker?

From my understanding, I consider Docker like a big ship. This ship can load a huge amount of containers on it, and each of them may contain different types of goods. They do not influence each other because they are individually blocked. Therefore, I do not need two ships to load food and chemistry products.

Docker does a similar thing. Here is an image from Docker official website showing how Docker works:

In this case, our containers for food and chemistry products become containers for different apps such as Node.js, MongoDB, etc. Each of them has its own dependencies, but they do not influence each other.

Why use Docker?

When coding for projects in university, sometimes the code works just fine on my machine, but if my teammates ran on theirs, there would be errors because of the environment settings. Therefore, it is always time-consuming and difficult to set up the same environment and configurations on each laptop.

Also, in a company, there are different environments as well:

  • Dev
  • Test
  • Production

Sometimes the code works perfectly on the dev environment, but it crashes on the other.

Therefore, suppose if the Development Team wants to deploy a web project in the past days, they have to set up environments for their database, front-end and server separately, and make sure there are no conflicts between them. After this, they need to test the web application in different environments to make sure it works fine there as well. If there is an error, they have to look back in and change the environment, and there is a high probability to affect other services. When deploying, they send the application to a different machine of the Operation Team, and the Dev Team has to teach them how to set up the environment again.

From the above example, we can see that the deployment process is super time-consuming, and the migration can be really difficult.

Also, people used a virtual machine to solve the conflict problem among different apps, which in this case means to virtualize a computer into multiple operating systems. Hence, different applications can run on different operating systems simultaneously without influencing each other. However, VMs always take a large amount of the HD space, and it takes minutes to bring up a VM.

In contrast, Docker can run on almost every system and platform. Once the configuration for the application is done in the container, it can be used on different machines without any errors. If there is an update to the dependencies, only a small amount of work is needed, which makes the management more efficient.

Meanwhile, Docker is usually brought up in seconds, and the containers are much smaller in size. It saves people more time.
